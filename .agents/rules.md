"Planning First" Mode: Always generate an Implementation Plan (an Artifact) before writing code. Prompt the user to review and approve the plan.
Context Pinning: If you are working on the Astro frontend, "Pin" the Hono API types file. This tells the Frontend Agent: "Whatever you build, it MUST match this file."
The "Flash" Rule: Use the Gemini 3 Flash model for the "Worker" agents (Frontend/Backend) to save money. Save the expensive Gemini 3 Pro model for the Orchestrator who has to do the heavy thinking.
"No Magic" Rule: Do not assume the user has run `wrangler login` or set up secrets. If you need a secret (like `BETTER_AUTH_SECRET`), you MUST explicitly tell the user to run `bun run deploy` (which runs `alchemy deploy`) to set it up first.
"Type Safety" Rule: When writing TypeScript, always use the types generated by Alchemy (found in `packages/env/env.d.ts`). Do not invent your own types for the Cloudflare environment. Always use TypeScript strict mode.
In apps/web/ Frontend Component Selection Rule: When generating or refactoring frontend components, follow this selection hierarchy:
  - Default to DaisyUI for "Visual-First" components. Use this for buttons, cards, badges, hero sections, and simple layouts where standard HTML/CSS behavior suffices. Use pure Tailwind classes and DaisyUI utility names.
  - Select Shadcn UI only when the component requires Advanced Primitive Logic. This is specifically for components where accessibility (ARIA) and complex state management are non-trivial.
    - Triggers for Shadcn: Form validation (with react-hook-form), complex Overlays (Dialogs/Sheets requiring focus traps), Data Tables, or nested Popovers.
  - The "State" Exception: Do not use Shadcn just because a component has state (e.g., a simple useState for a counter). Use Shadcn when the state impacts accessibility or keyboard navigation (e.g., selecting an item in a dropdown).

In apps/server/ (Backend Router): This is exclusively the routing and entry point layer (using Hono). Do NOT define complex business logic, database operations, or core authentication logic here. Instead, configure middleware (CORS, logging), define API routes, and import the core mechanics from the `@cornwall-ponds-gemini/*` packages.

In packages/ (Core Logic & Modules): This is where all backend mechanics belong. Keep packages isolated, modular, and focused:
  - `packages/auth`: Use this exclusively for Better Auth configuration, session management, and authentication providers. Do not add routing here.
  - `packages/db`: Use this exclusively for Drizzle database schemas, connections, and core queries. 
  - `packages/env`: Use this exclusively for environment variables and typing. Both `apps/server` and other packages should read secrets from here.
  - Cross-Dependency Rule: `apps/server/` can import from `packages/*`, and packages can import from other packages (e.g. `packages/auth` depends on `packages/db`), but `packages/*` must NEVER import anything from an `apps/` folder.
